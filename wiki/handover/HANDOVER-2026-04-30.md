## 🏗️ 核心架構決策

採用 **Karpathy LLM Wiki 模式**（非傳統 RAG），三層 repo 結構：

```
retail-iq/
├── raw/                     ← immutable 原始資料（LLM 唯讀）
│   └── <source>/<YYYY-Www>/*.html + *.meta.json
├── wiki/                    ← LLM 維護的結構化 markdown
│   ├── AGENTS.md            ← 規則與 schema（唯一人寫檔）
│   ├── index.md、log.md
│   ├── products/、concepts/、comparisons/、sales-playbook/
│   └── weekly-digest/
└── sources-config.yaml      ← 機器可讀來源設定
```

**核心取捨**：
- Git + Markdown（可讀、可審核、可 PR） vs 向量資料庫（快但不透明）
- Schema-first（先設計規則再寫內容）vs ad-hoc 填資料
- 分級來源（T1/T2/T3/T4）vs 混合所有來源同等對待

## ✨ 本週最重要的 5 個設計決策

1. **採用 Karpathy Wiki 模式，不採用純 RAG**
   - 理由：知識需可稽核、跨頁一致、compounding over time
   - 取捨：略高前期投資，長期維運成本顯著低於純 RAG

2. **Signature Demo 的三層結構（動作 / 口白 / 客戶看到）**
   - 這是「wiki → 腳本」可行的關鍵槓桿點
   - 驗證：三份 5 分鐘腳本都能順暢從此結構組合出來

3. **腳本長度三變體 + 草稿迭代（§4.3 + §4.4）**
   - 1 分鐘無 hands-on、純外觀 — 大幅降低門市現場心理門檻
   - Pass 1 純文字草稿讓使用者對內容有否決權
   - 符合 AI 產品「human in the loop」最佳實踐

4. **FAB + P 話術原則（§4.5）**
   - F/A/B/P 四層把規格轉為說服力話術
   - B 層 persona-adaptive，同 F+A 不同客群不同 B
   - P 層留空給店員填入親身故事 → 創造獨特性 + 記憶點

5. **來源 Tier 分級（T1–T4）**
   - 解決「官方 vs 第三方媒體 vs rumor」的合規兩難
   - URL pattern + 標題關鍵字 + Apple Newsroom cross-ref 三層過濾
   - 讓 MacRumors / 9to5Mac 的 review 部分可用，rumor 部分被阻擋

- 產品頁瀏覽（讀 markdown 渲染）

**預估時程**：2 週
**風險**：Magic Link 認證整合、PWA 在 iPad 現場的體驗調校

### Option C：補齊 W1 細節 📝
- 建立 1-2 個客群畫像頁（`business.md`、`student.md`）作範本
- 建立 1 個概念頁（`apple-intelligence.md`）示範跨產品引用

**預估時程**：3-5 天
**風險**：低，純內容補齊

**我的推薦**：**A → C → B**（先證明自動化可行，補齊內容骨架，再包前端給使用者用）
